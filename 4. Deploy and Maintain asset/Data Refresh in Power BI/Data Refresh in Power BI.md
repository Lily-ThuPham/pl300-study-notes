Ensuring data is recent is critical for delivering accurate results. This note covers the data refresh features, dependencies, and best practices.

### 1. Understanding Refresh by Storage Mode
The refresh requirements for a semantic model are entirely dependent on its **storage mode**.

| **Storage Mode**       | **How it Accesses Data**                                                                         | **Is Data Refresh Needed?**                                                                                                                            |
| ---------------------- | ------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Import** üì∏          | Power BI copies data from the source and stores it in an in-memory cache.                        | ‚úÖ **Yes (Essential)**. The model is a point-in-time copy. A data refresh is required to fetch changes from the underlying sources.                     |
| **DirectQuery** üì°     | Power BI sends queries directly to the underlying data source with every user interaction.       | ‚ùå **No**. The data is queried live and is always as current as the source. (Note: Dashboard tiles _do_ cache results, which refresh about every hour). |
| **Live Connection** üì∫ | Connects to an external model (e.g., SSAS, AAS, or another Power BI dataset).                    | ‚ùå **No (in Power BI)**. The refresh is managed by the _external_ source model, not within Power BI.                                                    |
| **Push** üì®            | Does not connect to a data source. Data is pushed into the model via an external service or API. | ‚ùå **No**. The data is refreshed by the external process that pushes it.                                                                                |
**Key Details for Import Mode:**
- A refreshed model is either a **full refresh** or a **partial refresh** (using an incremental refresh policy).
- During a full refresh, Power BI keeps a copy of the existing model to handle user queries. This means a full refresh requires **twice the amount of memory** the semantic model normally occupies.

### 3. Data Source Dependencies & Gateways
No refresh can succeed unless the Power BI service can access the data source.
- **Cloud Data Sources:** (e.g., Azure SQL DB, SharePoint Online)
    - **No gateway is required.** Power BI can connect directly.
    - Credentials are set in the dataset settings under **Data source credentials**.
- **On-Premises Data Sources:** (e.g., a SQL Server or file on a local network)
    - An **on-premises data gateway is mandatory** to act as a secure bridge.
    - An admin must add the data source definition (connection info, credentials) to the gateway
    - The dataset owner must be added as a **user** of that gateway data source.
    - The dataset owner must then **map** their dataset to the gateway in the dataset settings.
- **Mixed Sources (On-Premises + Cloud):**
    - If a single Power Query mashup query merges or appends data from an on-premises source and a cloud source, a **gateway is required for both**. The query for the cloud source will also be routed through the gateway.
    - To allow this, an admin must either add the cloud source to the gateway or check the box **"Allow user's cloud data sources to refresh through this gateway cluster."**
### 4. Configuring and Monitoring Refreshes
- **Scheduled Refresh:**
    - **Location:** Semantic model settings > **Refresh**.
    - **Frequency (Shared):** Max **8** refreshes per day.
    - **Frequency (Premium/PPU/Fabric):** Max **48** refreshes per day (and unlimited API refreshes).
    - **Time Limits:** Max **2 hours** per refresh on Shared; **5 hours** on Premium.
    - **Timing:** "Best effort," initiates within 15 minutes of the scheduled slot, but can be delayed up to 1 hour.
    - **Limit of Refresh failure**: after consecutive 4 failed refresh, the scheduled refresh will be disable. 
- **On-Demand Refresh:**
    - Triggered manually via **"Refresh now."**
    - Does **not** count toward the scheduled refresh limit.
![Refresh now](https://learn.microsoft.com/en-us/power-bi/connect-data/media/refresh-data/refresh-now.png)
- **Refresh History:**
    - Found in the dataset settings, this log shows the status (succeeded/failed) of past refreshes.
    - It is critical for troubleshooting, as it will show **multiple failed attempts** if Power BI retries a refresh.
    - It also breaks down the refresh into **Data** and **Query Cache** phases.
![Refresh history](https://learn.microsoft.com/en-us/power-bi/connect-data/media/refresh-data/refresh-history.png)
- **Fabric Monitoring Hub:** A new, centralized hub for monitoring refresh activities across all semantic models.
- **Failure Notifications:**
    - By default, an email is sent to the semantic model owner on failure. You can add other contacts to the notification list.
    - The schedule is **automatically deactivated** after **four consecutive failures** or after **two months of inactivity**.
### 5. Refresh Limitations
- **Dynamic Data Sources:** A refresh will fail if the connection info (e.g., server name, database, file path) is generated dynamically by another query. **However, refreshes _are_ supported if the query references Power Query parameters.**
- **Refresh Cancellation:** You can cancel a refresh in progress (for Import/Composite models) in Premium, PPU, or Fabric workspaces.7
### 6. Best Practices Summary
- Schedule refreshes for **less busy times** to avoid resource overload.8
- For large, slow-refreshing datasets, use **incremental refresh** instead of a full refresh.
- Optimize your model: only include tables and columns you actually need.
- Use separate gateways for Import and DirectQuery models to prevent scheduled refreshes from slowing down live reports.9
- Ensure **privacy settings** are manually re-applied in the service, as they do not publish from Desktop.
- Limit the number of visuals on dashboards, as a high number of tiles can increase refresh duration.
